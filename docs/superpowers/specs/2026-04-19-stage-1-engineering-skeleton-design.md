# 阶段 1：工程骨架设计

## 1. 背景

`WebToActions` 已完成 `阶段 P` 与 `阶段 0：技术 Spike`。当前仓库已经具备最小后端、最小前端和浏览器 PoC，但这些产物仍明显带有 Spike 语义：

- 后端仅有 `health` 与 `spike/context` 两个接口；
- 前端仅有单页健康检查验证；
- 工程目录尚未形成可持续扩展的正式结构；
- 文档口径与代码口径之间仍需要持续保持同步，避免阶段状态先于真实实现被提前回写。

因此，`阶段 1` 的目标不是重写现有 Spike，也不是提前实现 `阶段 2+` 功能，而是把已验证可行的最小链路收口为正式工程骨架，为后续领域模型、录制链路与审核链路提供稳定起点。

## 2. 目标

本阶段只解决“正式工程能否稳定起跑”的问题，目标是：

1. 统一后端正式工程入口与配置读取方式；
2. 统一前端页面入口、基础路由与一级导航壳子；
3. 统一开发期运行方式与文档口径；
4. 为后续 `session / recording / review / action / execution / importexport` 模块预留清晰目录边界；
5. 保持对现有 Spike 产物的复用，不推翻已经验证通过的 PoC。

## 3. 非目标

本阶段明确不做以下内容：

- 不实现 `SQLite` 表结构、仓储层和迁移机制；
- 不实现录制、审核、执行的真实业务接口；
- 不引入 `SSE` 真实事件流；
- 不构建完整的 `ActionMacro`、`MetadataDraft`、执行中心等业务闭环；
- 不为了“看起来完整”而提前铺设大量空实现。

换句话说，`阶段 1` 的交付边界是“可运行、可扩展、可接力”，不是“业务功能完整”。

## 4. 备选方案与决策

### 方案 A：最小正式收口（选定）

在现有 Spike 基础上，仅补齐正式工程骨架所必需的配置、目录、页面壳子、启动方式和文档同步。

优点：

- 与主执行文档的阶段边界最一致；
- 返工最少；
- 能最快进入 `阶段 2`。

缺点：

- 页面与模块占位会较为朴素；
- 不会一次性拥有“看起来很完整”的工程外观。

### 方案 B：强化骨架

在方案 A 基础上，提前引入更多基础设施和占位实现，例如完整页面树、组件库、更多目录和更重的前端数据层。

优点是结构更饱满；缺点是容易把 `阶段 1` 与 `阶段 2+` 混在一起，增加样板代码和后续返工。

### 方案 C：保守修补

只修复现有明显问题，例如文档口径、Python 版本说明、前端重复 HTML 内容等，不主动建立正式骨架。

优点是改动最小；缺点是后续阶段仍要回头补基础设施，不利于连续开发。

### 决策

选定 **方案 A：最小正式收口**。后续所有实现都应以“刚好满足 `阶段 1` 验收要求”为边界。

## 5. 总体设计

### 5.1 后端设计

后端从 Spike 入口升级为正式应用入口，但仍保持轻量：

- 保留 `FastAPI` 作为应用壳子；
- 新增 `core/config` 统一读取应用名、环境、数据目录、前端开发源地址、浏览器默认策略等基础配置；
- `main.py` 只负责创建应用、装配路由、暴露应用实例；
- `api/routes/health.py` 从“Spike 状态展示”调整为“正式骨架健康检查 + 基础运行时信息”；
- 先创建后续模块包目录和 `__init__.py`，但不写入未被使用的空服务实现。

设计原则是：让 `阶段 2` 的领域模型和存储代码可以直接落进既定目录，而不是继续堆在根入口周围。

### 5.2 前端设计

前端从单页 Spike 升级为最小正式管理台壳子：

- 引入 `React Router` 作为正式路由入口；
- 首页保留健康检查能力，作为“系统状态概览页”；
- 增加一级导航占位，至少体现首页、录制中心、会话管理、审核、动作、执行、导入导出等后续入口；
- 补齐 `router / components / features / types` 目录；
- API 调用继续通过 `services/` 封装，避免页面直接发请求。

本阶段不强制引入 `TanStack Query` 和 `Ant Design` 的完整用法，但目录和页面组织必须为后续接入留好边界。

### 5.3 运行方式设计

运行方式按“开发期前后端分离，运行期后端单入口挂载”的技术方案保持不变。本阶段先完成开发期清晰化：

- 后端提供明确启动命令；
- 前端提供明确启动命令；
- 文档中显式标注 `Python 3.11+` 为正式工程基线；
- 对本机当前仍是 `Python 3.10.0` 的事实保留说明，但不再以其作为正式兼容目标。

本阶段将明确新增：

- `scripts/dev/start_backend.sh`
- `scripts/dev/start_frontend.sh`

这样既满足“建立本地运行脚本”的阶段要求，也能为后续新 `session` 提供稳定入口。

### 5.4 文档与阶段状态同步

本阶段设计要求同步维护以下文档状态：

- `docs/技术文档/开发步骤拆解.md`：将 `阶段 1` 标记为 `进行中`；
- `README.md`：从“下一步进入阶段 0”更新为“当前正在阶段 1”；
- `docs/superpowers/plans/2026-04-19-stage-1-engineering-skeleton.md`：维护阶段 1 的实施步骤与验证方式；
- 不再额外维护独立的过程日志文件，分析、结论和状态应直接回写到长期文档。

这样做的目的是保证新 `session` 无需重新判断“现在到底做到哪里”。

## 6. 组件边界

### 6.1 后端组件

- `app/main.py`：应用工厂与应用装配；
- `app/core/config.py`：环境变量与默认值解析；
- `app/api/routes/health.py`：骨架阶段的健康检查与基础运行时信息；
- `app/session/`、`app/recording/`、`app/review/`、`app/action/`、`app/execution/`、`app/importexport/`、`app/browser/`、`app/infrastructure/`：仅建立包边界，不提前注入业务实现。

### 6.2 前端组件

- `src/router/`：路由定义与导航模型；
- `src/pages/`：页面级容器；
- `src/components/`：最小共享导航或页面壳子；
- `src/services/`：健康检查等 API 调用；
- `src/features/`、`src/types/`：先建立边界，逐步填充。

## 7. 数据流

本阶段仅涉及最小数据流：

1. 用户打开前端首页；
2. 前端路由渲染首页壳子；
3. 用户触发健康检查；
4. 前端通过 `services/health` 调用后端；
5. 后端通过正式应用入口返回当前健康信息与基础运行时元数据；
6. 前端展示结果。

该数据流足以证明：

- 前后端工程壳子已就位；
- 正式路由结构可工作；
- 统一配置与统一 API 入口已生效。

## 8. 错误处理

本阶段只做最小错误处理收口：

- 后端健康检查接口保持稳定 JSON 结构，不混入多形态响应；
- 前端首页对健康检查失败给出明确提示；
- 配置读取使用默认值 + 环境变量覆盖，避免因缺少非关键配置而直接崩溃；
- 若当前环境不满足 `Python 3.11+`，文档明确提示，而不是在代码中隐式兼容或静默忽略。

## 9. 测试策略

本阶段测试遵循“只为新增正式行为补最小验证”：

- 后端：对应用配置与健康检查返回结构补自动化测试；
- 前端：补最小 `Vitest + React Testing Library` 测试基础设施，并对基础路由壳子和首页渲染补最小测试；
- 手工验证：运行前后端后，确认首页可访问且健康检查正常。

测试范围必须服务于 `阶段 1` 的骨架目标，不为未来功能提前堆测试。

## 10. 验收标准

`阶段 1` 视为完成，需要满足以下条件：

1. 后端具备正式应用入口和配置读取模块；
2. 前端具备正式路由壳子和一级导航占位；
3. 前后端本地联调仍可完成健康检查；
4. 文档口径与代码状态一致；
5. 新 `session` 进入仓库后，可直接从正式骨架继续进入 `阶段 2`，无需重新理解 Spike 杂糅结构。

## 11. 风险与约束

- 当前本机默认 `python3` 仍为 `3.10.0`，会影响本地即时验证；需要在实现时把“目标基线”和“本机现状”区分清楚。
- 当前后端测试环境尚未准备完毕，正式实现时可能需要先补虚拟环境和依赖安装步骤。
- 前端当前已经可构建，任何阶段 1 改造都不应破坏这一基线。

## 12. 实施后的下一步

当本设计落地并通过验证后，下一步应进入 `阶段 2：领域模型与存储骨架`，而不是继续在 `阶段 1` 内追加更多占位实现。
