# 开发规范

## 文档归类

- 当前归类：技术文档
- 技术文档目录：[文档索引](./文档索引.md)
- 配套进度文档：[开发步骤拆解](./开发步骤拆解.md)

## 1. 文档目的

本文档不是完整的“企业级研发规范大全”，而是为了支撑当前 `阶段 0：技术 Spike` 和后续 `阶段 1：工程骨架` 的最小开发约束。

目标只有三个：

- 先保证目录和分层不会从第一天就失控；
- 先保证实验代码与正式代码边界清楚；
- 先保证后续进入正式开发时，不需要推翻当前 `Spike` 结果。

## 2. 当前适用范围

当前规范优先适用于：

- `阶段 0：技术 Spike`
- `阶段 1：工程骨架`

在 `Spike` 结束前，不写过细的页面规范、组件抽象准则和复杂状态管理规范。  
这些内容应在验证完浏览器链路、前后端联调和最小动作闭环后，再补成正式规范。

## 3. 工具链基线

### 3.1 目标技术栈

- 后端：`Python 3.11+`
- API：`FastAPI`
- 浏览器控制：`Playwright Python`
- 前端：`React + TypeScript + Vite`
- 前端数据获取：`TanStack Query`
- 组件库：`Ant Design`

### 3.2 当前本机约束

当前本机 `python3` 版本为 `3.10.0`。因此：

- `Spike` 代码需尽量兼容 `Python 3.10`；
- 不使用只在 `Python 3.11+` 才可用的语法特性；
- 在进入正式工程化阶段前，应切换到 `Python 3.11+` 环境。

### 3.3 依赖管理约束

- 后端依赖放在 `backend/pyproject.toml`；
- 前端依赖放在 `frontend/package.json`；
- 不把临时脚本依赖写到 README 或命令备注中，必须落到依赖清单；
- `Spike` 新增依赖必须直接服务于验证目标，避免提前引入无关库。

### 3.4 浏览器运行约束

- 默认验证基线使用 `Playwright` 管理的 `Chromium`；
- 允许通过 `channel="chrome"` 之类方式使用本机已安装浏览器做兼容性验证；
- 只有标准 channel 无法满足时，才引入自定义浏览器可执行路径；
- “更换浏览器程序”和“复用用户日常 Profile / 登录态”是两件事，当前阶段不要混为一谈；
- 在进入正式工程化前，浏览器启动策略应统一收口到桥接层，不要散落在各处脚本或路由中。

## 4. 目录与分层约束

## 4.1 后端目录

后端按“正式代码”和“实验代码”分开：

- `backend/app/`：后续可持续演进的正式代码；
- `backend/tests/`：测试代码；
- `backend/spikes/`：阶段 0 的独立 PoC 脚本。

约束如下：

- API 路由只负责参数接收、响应拼装和错误映射；
- 业务流程编排放在应用层；
- 浏览器控制能力不能直接散落在路由里；
- 实验性验证脚本优先放在 `backend/spikes/`，不要污染正式模块。

### 4.2 前端目录

前端先按以下最小结构组织：

- `src/pages/`：页面级容器
- `src/features/`：业务模块
- `src/components/`：通用或跨页面组件
- `src/services/`：接口调用和事件流封装
- `src/types/`：共享类型

约束如下：

- 页面组件不直接手写 `fetch`；
- 请求封装统一放在 `services/`；
- 服务器数据优先通过 `TanStack Query` 管理；
- 与录制、审核、执行强相关的 UI 先放在 `features/`，不要过早沉淀到全局组件层。

## 5. 命名与代码风格

### 5.1 Python

- 使用类型标注；
- 新增函数优先保持短小，单一职责；
- 日志统一使用 `logging`；
- 配置统一从配置模块读取，不在业务代码里硬编码环境变量；
- 能抽成纯函数的逻辑，尽量不要直接绑在框架对象上。

### 5.2 TypeScript / React

- 使用函数组件；
- 组件命名使用 `PascalCase`；
- Hook 命名使用 `useXxx`；
- 页面组件负责组织布局，复杂交互放到 `features/`；
- 避免把接口返回结构直接散落到页面中，必要时在 `types/` 定义视图模型。

## 6. 接口与事件约束

### 6.1 HTTP API

- 查询类接口保持简单 REST 风格；
- 健康检查、配置读取、录制启动、录制结束等接口应保持职责单一；
- 返回结构尽量稳定，避免同一接口返回多种形态。

### 6.2 SSE

`SSE` 只承接长任务状态，不承接普通查询。

当前阶段约束：

- 每个事件至少包含 `event`、`runId`、`status`、`timestamp`；
- 事件负载结构保持可扩展，不为首版过度设计；
- `Spike` 阶段可以先验证最小事件流，不要求一次覆盖所有业务场景。

## 7. 数据与文件落盘约束

- 结构化元数据和临时索引，后续统一收口到 `SQLite`；
- 原始证据、导出包和调试产物优先放文件系统；
- `Spike` 阶段允许先写 JSON 文件，但目录结构要为后续工程化留口；
- 任何录制证据都必须区分“原始抓取结果”和“后续整理结果”。

建议当前阶段的实验目录：

- `.webtoactions/spikes/recordings/`
- `.webtoactions/spikes/executions/`
- `.webtoactions/spikes/tmp/`

## 8. 测试与验证约束

- 正式可复用逻辑优先补最小自动化测试；
- `Spike` 脚本允许使用手工验证，但验证命令和结论必须回写文档；
- 健康检查、配置装载、纯函数转换逻辑这类稳定代码，优先补测试；
- 浏览器 PoC 的重点是“能否证明链路可行”，不是在 `阶段 0` 就追求高覆盖率。

## 9. Spike 阶段特别约束

- 这一阶段的目标是“验证可行性”，不是“把正式系统提前做完”；
- 只验证当前阶段文档明确要求的链路；
- 不提前实现 `DOM` 细粒度事件轨迹；
- 不提前实现完整审核台、动作库和导入导出；
- 默认先验证隔离上下文，不直接绑定用户日常浏览器主 `Profile`；
- 任一 PoC 一旦证明不可行，优先记录原因和替代方案，而不是继续堆实现。

## 10. 进入正式开发前需要补齐的规范

完成 `阶段 0` 后，应根据真实验证结果补充：

- 前端组件分层和复用规范；
- 状态管理与 `TanStack Query` 使用规范；
- 后端模块边界与错误码规范；
- `SSE` 事件类型清单；
- 存储目录约定和迁移策略；
- 测试分层规则。
