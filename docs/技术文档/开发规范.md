# 开发规范

## 文档归类

- 当前归类：技术文档
- 技术文档目录：[文档索引](./文档索引.md)
- 配套进度文档：[开发步骤拆解](./开发步骤拆解.md)

## 1. 文档目的

本文档用于约束当前仓库进入稳定开发后的最小工程规范，服务已经完成的 `阶段 1-7` 基线，以及后续迭代开发。

目标有三个：

- 保持后端、前端、存储和文档边界稳定，不因为新增功能重新打散结构；
- 让新增能力优先复用既有“录制 -> 审核 -> 动作 -> 执行 -> 导入导出”的主闭环；
- 让代码、测试、README 与阶段文档始终保持同一套事实口径。

## 2. 当前适用范围

当前规范适用于：

- 已完成的 `阶段 1-7` 能力维护、缺陷修复与扩展；
- 后续在 `BusinessAction`、真实浏览器 E2E、CI 集成、导入兼容策略等方向上的迭代；
- 当前仓库与独立 `worktree` 的日常开发。

以下早期约束已不再作为当前基线：

- 不再按 `Spike / 阶段 1` 语境限制实现；
- 不再要求兼容 `Python 3.10`；
- 不再把 `TanStack Query`、`Ant Design` 视为既定技术选型。

## 3. 工具链基线

### 3.1 当前技术栈

- 后端：`Python 3.11+`
- API：`FastAPI`
- 浏览器控制：`Playwright Python`
- 前端：`React + TypeScript + Vite + React Router`
- 本地存储：`SQLite + 文件对象区`
- 长任务状态：`SSE`
- 自动化验证：`pytest + vitest`

### 3.2 运行与环境基线

- 当前开发必须使用 `Python 3.11+` 虚拟环境，不再兼容 `Python 3.10`；
- 开发模式保持前后端分离，运行模式保持后端挂载前端 `dist` 的单进程入口；
- 默认数据目录仍为 `.webtoactions/`，但所有路径应通过配置模块解析，不在业务代码里硬编码；
- 端口占用属于常见本地环境问题，启动脚本与文档必须保留端口覆盖说明。

### 3.3 依赖管理约束

- 后端依赖统一放在 `backend/pyproject.toml`；
- 前端依赖统一放在 `frontend/package.json`；
- 新增依赖必须直接服务当前需求，不为“以后可能会用”提前引库；
- 不把依赖安装步骤散落在聊天结论或注释里，必须同步到依赖清单与正式文档。

### 3.4 浏览器与会话约束

- 默认稳定基线仍为 `Playwright` 管理的隔离 `Chromium`；
- 允许通过浏览器 channel 使用本机已安装浏览器做兼容验证；
- “切换浏览器程序”和“复用用户日常登录态”是两件事，不能混为一谈；
- 导入资料包时不能恢复活跃登录态，导入后的会话必须明确降级为需重新登录。

## 4. 目录与分层约束

### 4.1 后端

- `backend/app/api/` 只负责参数接收、响应拼装和错误映射；
- `backend/app/*/service/` 负责业务流程编排，不把流程直接塞进路由；
- `backend/app/*/domain/` 负责领域对象与状态约束；
- `backend/app/infrastructure/` 负责 SQLite、存储布局等基础设施；
- `backend/tests/` 保持按能力分层组织，不把验证逻辑写回正式代码；
- `backend/spikes/` 仅保留实验性 PoC，不与正式链路耦合。

### 4.2 前端

- `src/pages/` 放页面级容器；
- `src/features/` 放录制、审核、执行等复杂交互；
- `src/components/` 放跨页面复用组件；
- `src/services/` 统一封装 `fetch`、下载上传与 `SSE`；
- `src/types/` 维护前后端契约对应类型。

约束如下：

- 页面组件不直接手写裸 `fetch`；
- 接口错误解析优先在 `services/` 层统一处理；
- 长任务事件流统一走 `services/events/` 封装；
- 共享错误态、空态等基础表现应复用，不重复堆内联样式与文案。

## 5. 命名与代码风格

### 5.1 Python

- 使用类型标注；
- 函数尽量保持短小、单一职责；
- 配置统一从配置模块读取；
- 能抽成纯函数的逻辑，尽量不要绑在框架对象上；
- 版本化对象优先保持不可变更新，而不是到处原地修改字段。

### 5.2 TypeScript / React

- 使用函数组件；
- 组件命名使用 `PascalCase`；
- Hook 命名使用 `useXxx`；
- 页面负责组织布局，复杂状态与订阅逻辑优先下沉到 `features/` 或 `services/`；
- 避免把接口原始结构直接散落到页面中，必要时先在 `types/` 中收口。

## 6. 接口、事件与错误约束

### 6.1 HTTP API

- 查询与命令接口保持稳定契约，不让同一路由返回多套形态；
- 失败场景优先返回明确 `detail`，不要把可解释错误吞成泛化 500；
- 导入导出、执行启动、审核保存等高风险路径必须覆盖 4xx/409/422 等失败分支。

### 6.2 SSE

- `SSE` 只承接录制、审核、执行等长任务状态，不承接普通列表查询；
- 事件载荷必须可直接驱动前端刷新，不依赖隐式上下文；
- 订阅断开时，前端应能回退到重新拉取详情或重新订阅，而不是静默失效。

## 7. 数据与文件落盘约束

- 结构化元数据进入 `SQLite`；
- 请求体、响应体、会话快照、执行日志、导出包等大对象进入文件对象区；
- 数据目录结构以 `.webtoactions/evidence`、`.webtoactions/actions`、`.webtoactions/runs`、`.webtoactions/exports` 为准；
- 任何导入流程都要先校验资料包完整性，再落盘保存，避免半导入；
- 任何导出流程都要保证资料包内部自洽，不能漏掉被执行记录引用的历史版本工件。

## 8. 测试与验证约束

- 后端行为改动优先补 `pytest`；
- 前端交互与错误提示改动优先补 `vitest`；
- 涉及运行模式、构建链路或静态资源挂载时，必须额外执行真实构建或 smoke 验证；
- 在声称“完成”“通过”“修复”前，必须重新跑对应验证命令并检查输出；
- 手工冒烟路径统一维护在 `docs/技术文档/手工冒烟清单.md`，不再依赖独立的过程记录文件。

## 9. 文档同步约束

- 阶段状态变化至少同步到 `README.md`、`docs/技术文档/开发步骤拆解.md` 与受影响的正式设计文档；
- 启动方式变化同步到 `docs/本地启动说明.md`；
- 不再维护 `task_plan.md`、`progress.md`、`findings.md` 这类独立过程文件，过程结论应沉淀到正式文档；
- 若代码、测试与文档冲突，先以仓库事实为准，再回写文档。
