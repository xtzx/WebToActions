## 1. 文档目的

本文档用于把首版开发过程拆成若干可控阶段，让项目始终按“先验证关键路径，再扩展功能”的方式推进。

它和 [首版实现计划](./首版实现计划.md) 的区别是：

- 本文档强调阶段推进和验收顺序；
- `首版实现计划` 更偏工程落地与模块任务清单。

## 2. 总体推进原则

- 始终优先打通用户闭环，而不是优先做技术层完备性。
- 每一阶段完成后都要能演示或验证。
- 前一个阶段未稳定，不进入下一个阶段。
- 对核心不确定点优先做技术 `Spike`。
- 首版不追求一步到位，先完成 `ActionMacro` 闭环。

## 3. 推荐阶段总览

```mermaid
flowchart LR
Spike[阶段0 技术Spike] --> Skeleton[阶段1 工程骨架]
Skeleton --> Models[阶段2 领域与存储]
Models --> Recording[阶段3 录制链路]
Recording --> Review[阶段4 审核链路]
Review --> Macro[阶段5 动作宏与执行]
Macro --> ImportExport[阶段6 导入导出]
ImportExport --> Hardening[阶段7 稳定性与收尾]
```

## 4. 阶段 0：技术 Spike

### 目标

先验证最不确定的技术链路，而不是马上搭完整工程。

### 要做的事

- 验证 `Playwright Python` 创建独立浏览器上下文；
- 验证页面导航和请求响应采集；
- 验证会话状态摘要是否能读取；
- 验证 `FastAPI` 与 `React` 本地联调；
- 验证一个最简单的宏执行闭环。

### 交付物

- 若干独立 PoC 脚本；
- 技术结论记录；
- 需要规避的坑清单。

### 验收标准

- 能打开指定 `URL`；
- 能抓到请求响应；
- 能拿到会话上下文摘要；
- 前后端本地联调成功；
- 至少一个最小动作链路被证明可行。

### 当前执行结果（2026-04-19）

- 已完成最小后端骨架：`backend/app/main.py` 与 `backend/app/api/routes/health.py`，并通过 `backend/tests/api/test_health.py`。
- 已完成最小前端骨架：`frontend/` 目录可构建，且通过 `Vite` 代理访问 `/api/health` 成功打到本地后端。
- 已完成浏览器 PoC：`backend/spikes/browser_recording_poc.py` 可真实打开 `https://example.com`，生成包含请求摘要、导航事件、`cookie`、`localStorage`、`sessionStorage` 摘要的 JSON 文件。
- 已验证最小闭环：`打开 URL -> 等待页面收口 -> 写入 JSON 证据` 可行；带 `--headed --hold-open-ms` 时可支持人工触发最小动作后再收口。
- 已补充浏览器启动策略：默认使用 `Playwright Chromium`，并支持 `--browser-channel chrome` 和 `--browser-path` 作为可选通道。

### 本阶段发现的坑

- `Playwright` 的页面状态、`storage_state()` 与页面存储快照必须在浏览器上下文关闭前采集，否则会出现 `Event loop is closed`。
- 本机已有其他服务占用常见开发端口，`Vite` 会自动切换端口；后续正式开发需要在启动脚本或配置里显式处理端口占用与发现。
- 当前本机 `python3` 仍为 `3.10.0`，虽然本次 `Spike` 可运行，但进入正式工程化前仍建议切换到 `Python 3.11+`。

### 完成后再进入下一阶段

只有 `Spike` 证明主技术链可行，才值得开始正式工程搭建。

### 本阶段发现的坑

- 浏览器会话实体在 `SQLite` 中被 `Recording` 外键引用后，更新会话状态不能再使用“先删后插”，必须改为真正的更新或插入。
- `SSE` 在测试环境里需要提供“只返回最新快照”的收口方式，否则 `TestClient` 的流式连接容易长时间挂住，导致阶段验证不稳定。
- 录制中页面的实时计数适合走内存态 `SSE` 推送，而不是在每个请求事件上都回写数据库；持久化应留在结束录制后的统一收口步骤完成。

## 8. 阶段 4：元数据审核 MVP

### 目标

完成“从录制结果到审核结果”的第二条闭环。

### 要做的事

- 建立 `MetadataDraft` 生成逻辑；
- 明确录制结束后只投递分析任务，不在录制链路中同步生成草案；
- 建立审核页；
- 建立元数据分析任务状态的 `SSE` 推送；
- 支持关键请求标注；
- 支持噪音请求标注；
- 支持参数来源确认；
- 支持字段说明编辑；
- 支持保存 `ReviewedMetadata` 新版本。

### 交付物

- 审核页面；
- 审核结果版本；
- 参数建议交互。

### 验收标准

- 用户可以从录制详情跳转到审核页；
- 用户可以完成一次审核并保存结果；
- 保存后可回看历史审核版本。
- 分析任务状态和完成结果可被前端实时更新。

### 当前执行结果（2026-04-19）

- 已完成后端阶段 4 实现：新增 `MetadataAnalysisService`、`ReviewJobRunner`、`ReviewService`，并提供 `/api/reviews/{recording_id}` 审核上下文 API、分析触发 API、审核状态 `SSE` 与审核保存 API。
- 已完成阶段 4 review 领域收口：`MetadataDraft` 现可持久化结构化参数建议与动作片段建议，`ReviewedMetadata` 新增噪音请求标注字段，`Alembic` 已补齐阶段 4 增量迁移。
- 已完成前端阶段 4 页面接入：新增 `/review/:recordingId` 审核页、请求审核面板、参数建议面板与审核状态流 hook，并从录制详情页补入“进入审核”入口。
- 已完成阶段 4 健康契约同步：`/api/health` 的 `phase` 已更新为 `stage4`，首页与导航文案也已切到元数据审核 MVP 语境。
- 已完成阶段 4 自动化验证：后端测试 `31 passed`、前端测试 `7 passed`、前端构建通过。

### 本阶段发现的坑

- `SQLite` 虽然支持 `ALTER TABLE ... ADD COLUMN`，但不支持通用的 `ALTER COLUMN DROP DEFAULT`；阶段 4 迁移必须按 `SQLite` 语法能力收口。
- 审核状态流的 `SSE` 快照只能告诉前端“分析已完成”，不能代替重新拉取完整审核上下文；前端在收到 `completed` 事件后仍需要回源 `GET /api/reviews/{recording_id}`。
- 托管推进时如果直接从录制详情跳到审核页，测试环境里要先等待录制详情真正加载完成，再触发“进入审核”，否则容易在前端测试里出现假阴性。

- 已完成资料包契约收口：资料包覆盖录制聚合、审核结果、关联动作宏、关联执行记录和被引用证据文件；导入后明确把浏览器会话标记为 `relogin_required`，不迁移活跃登录态。
- 已完成前端阶段 6 页面接入：`/importexport` 已替换为真实页面，可选择录制导出资料包、上传资料包导入，并展示导入警告。
- 已完成运行模式收口：后端支持挂载前端 `dist`，支持深层路由 SPA fallback，并提供 `scripts/run/start_local_app.sh` 单进程启动脚本。
- 已完成阶段 6 自动化验证：`backend/tests/importexport/test_importexport_flow.py`、`backend/tests/smoke/test_smoke.py` 通过，前端 `npm run build` 通过。

### 本阶段发现的坑

- 单进程运行模式必须阻止 `/api/*` 缺省回退到前端 `index.html`，否则未知 API 路由会被错误吞掉。
- 资料包导入时如果直接复用浏览器会话 ID 却保留登录态摘要，会给用户造成“可直接继续执行”的误导，因此要显式降级为 `relogin_required`。
- 本机常见端口可能已被占用，单进程运行脚本需要支持 `APP_PORT` 覆盖以便本地验证。

## 12. 每阶段的停顿点

建议在以下节点强制停下来复盘：

- `Spike` 完成后
- 录制链路打通后
- 审核链路打通后
- 执行链路打通后
- 导入导出完成后

每个停顿点都要回答：

- 当前最脆弱的链路是什么；
- 有没有必要提前引入更细的证据能力；
- 是否有阶段性范围失控；
- 下一阶段的前置条件是否真的满足。

## 13. 建议的实现顺序

严格建议按以下优先级执行：

1. 技术 `Spike`
2. 工程骨架
3. 领域模型与存储骨架
4. 录制链路
5. 审核链路
6. `ActionMacro` 执行
7. 导入导出
8. 稳定性收尾

不要一开始就先做：

- `BusinessAction` 完整闭环；
- 复杂动作编排；
- 细粒度 `DOM` 轨迹；
- 多人协作。

## 15. 当前建议

当前阶段计划已全部完成。如果马上进入新的开发 `session`，建议优先从以下后续方向里选一项继续：

> 1. 真实浏览器站点的全自动 E2E 与 CI 接入；  
> 2. `BusinessAction` 抽象与更高层业务编排；  
> 3. 资料包跨版本兼容、冲突合并与更细粒度导入策略。

这样既能延续当前阶段 7 的稳定基线，也不会把精力重新耗在已经收口完成的阶段上。
