新 `session` 建议遵循以下规则：

- 先看顶部“进度计划（执行看板）”，从第一个 `待开始` 阶段接着做；
- 当前不要重做 `阶段 0`，而是从 `阶段 1：工程骨架` 开始；
- 开始某阶段前，先把该阶段状态改成 `进行中`；
- 完成某阶段后，回写本文档、`findings.md` 与 `progress.md`；
- 发现方案边界变化时，先更新文档，再继续实现；
- 已有 `Spike` 产物只能复用和工程化，不要无理由推翻重写。

- 前端最小骨架已存在：`frontend/` 已可构建，并已验证通过代理访问 `/api/health`；
- 浏览器 PoC 已存在：`backend/spikes/browser_recording_poc.py`；

## 1. 文档目的

本文档用于把首版开发过程拆成若干可控阶段，让项目始终按“先验证关键路径，再扩展功能”的方式推进。

它和 [首版实现计划](./首版实现计划.md) 的区别是：

- 本文档强调阶段推进和验收顺序；
- `首版实现计划` 更偏工程落地与模块任务清单。

## 2. 总体推进原则

- 始终优先打通用户闭环，而不是优先做技术层完备性。
- 每一阶段完成后都要能演示或验证。
- 前一个阶段未稳定，不进入下一个阶段。
- 对核心不确定点优先做技术 `Spike`。
- 首版不追求一步到位，先完成 `ActionMacro` 闭环。

## 3. 推荐阶段总览

```mermaid
flowchart LR
Spike[阶段0 技术Spike] --> Skeleton[阶段1 工程骨架]
Skeleton --> Models[阶段2 领域与存储]
Models --> Recording[阶段3 录制链路]
Recording --> Review[阶段4 审核链路]
Review --> Macro[阶段5 动作宏与执行]
Macro --> ImportExport[阶段6 导入导出]
ImportExport --> Hardening[阶段7 稳定性与收尾]
```

## 4. 阶段 0：技术 Spike

### 目标

先验证最不确定的技术链路，而不是马上搭完整工程。

### 要做的事

- 验证 `Playwright Python` 创建独立浏览器上下文；
- 验证页面导航和请求响应采集；
- 验证会话状态摘要是否能读取；
- 验证 `FastAPI` 与 `React` 本地联调；
- 验证一个最简单的宏执行闭环。

### 交付物

- 若干独立 PoC 脚本；
- 技术结论记录；
- 需要规避的坑清单。

### 验收标准

- 能打开指定 `URL`；
- 能抓到请求响应；
- 能拿到会话上下文摘要；
- 前后端本地联调成功；
- 至少一个最小动作链路被证明可行。

### 当前执行结果（2026-04-19）

- 已完成最小后端骨架：`backend/app/main.py` 与 `backend/app/api/routes/health.py`，并通过 `backend/tests/api/test_health.py`。
- 已完成最小前端骨架：`frontend/` 目录可构建，且通过 `Vite` 代理访问 `/api/health` 成功打到本地后端。
- 已完成浏览器 PoC：`backend/spikes/browser_recording_poc.py` 可真实打开 `https://example.com`，生成包含请求摘要、导航事件、`cookie`、`localStorage`、`sessionStorage` 摘要的 JSON 文件。
- 已验证最小闭环：`打开 URL -> 等待页面收口 -> 写入 JSON 证据` 可行；带 `--headed --hold-open-ms` 时可支持人工触发最小动作后再收口。
- 已补充浏览器启动策略：默认使用 `Playwright Chromium`，并支持 `--browser-channel chrome` 和 `--browser-path` 作为可选通道。

### 本阶段发现的坑

- `Playwright` 的页面状态、`storage_state()` 与页面存储快照必须在浏览器上下文关闭前采集，否则会出现 `Event loop is closed`。
- 本机已有其他服务占用常见开发端口，`Vite` 会自动切换端口；后续正式开发需要在启动脚本或配置里显式处理端口占用与发现。
- 当前本机 `python3` 仍为 `3.10.0`，虽然本次 `Spike` 可运行，但进入正式工程化前仍建议切换到 `Python 3.11+`。

### 完成后再进入下一阶段

只有 `Spike` 证明主技术链可行，才值得开始正式工程搭建。

## 5. 阶段 1：工程骨架

### 目标

建立可运行的前后端工程基础。

### 要做的事

- 在现有 `Spike` 代码上收口正式后端工程骨架，而不是重新创建空工程；
- 在现有 `Spike` 代码上收口正式前端工程骨架，而不是重新创建空工程；
- 建立正式目录结构与基础分层；
- 建立本地运行脚本；
- 建立首页和基本导航；
- 建立 API 健康检查与基础配置读取；
- 建立基础测试框架；
- 为下一阶段预留 `session / recording / review / action / execution` 模块骨架。

### 交付物

- 后端服务可启动；
- 前端开发服务器可启动；
- 首页能访问；
- 前后端联通；
- 目录结构和基础分层可支持后续阶段直接继续开发。

### 验收标准

- 本地一条命令能分别启动前端和后端；
- 浏览器能访问首页；
- 前端可以调用健康检查接口。
- 新 `session` 进入仓库后，无需重新理解 `Spike` 脚本即可继续后续阶段。

## 8. 阶段 4：元数据审核 MVP

- 建立审核页；

- 支持关键请求标注；
- 支持噪音请求标注；
- 支持参数来源确认；
- 支持字段说明编辑；
- 支持保存 `ReviewedMetadata` 新版本。

### 交付物

- 审核页面；
- 审核结果版本；
- 参数建议交互。

### 验收标准

- 用户可以从录制详情跳转到审核页；
- 用户可以完成一次审核并保存结果；
- 保存后可回看历史审核版本。

## 12. 每阶段的停顿点

建议在以下节点强制停下来复盘：

- `Spike` 完成后
- 录制链路打通后
- 审核链路打通后
- 执行链路打通后
- 导入导出完成后

每个停顿点都要回答：

- 当前最脆弱的链路是什么；
- 有没有必要提前引入更细的证据能力；
- 是否有阶段性范围失控；
- 下一阶段的前置条件是否真的满足。

## 13. 建议的实现顺序

严格建议按以下优先级执行：

1. 技术 `Spike`
2. 工程骨架
3. 领域模型与存储骨架
4. 录制链路
5. 审核链路
6. `ActionMacro` 执行
7. 导入导出
8. 稳定性收尾

不要一开始就先做：

- `BusinessAction` 完整闭环；
- 复杂动作编排；
- 细粒度 `DOM` 轨迹；
- 多人协作。

## 14. 与首版实现计划的关系

如果你要进入真正编码阶段，建议：

- 先按本文决定“先做哪个阶段”；
- 再参考 [首版实现计划](./首版实现计划.md) 的文件级落地建议；
- 编码时一次只执行一个阶段，不要把多个阶段混在一起。

## 15. 当前建议

如果马上进入新的开发 `session`，最合理的起点不是重做 `Spike`，而是：

> 直接从 **阶段 1：工程骨架** 开始，在现有 `Spike` 产物基础上做工程化收口；  
> 完成后再进入 **阶段 2：领域模型与存储骨架**。

这样既不会丢掉已经验证通过的技术结论，也能让新 `session` 直接进入真正有产出的开发阶段。
