后续开发推进时，以本节状态为准。

| 阶段 | 名称 | 当前状态 | 说明 |
|------|------|----------|------|
| P | 开发前置 | `已完成` | 已完成仓库初始化、基础文件补齐与首个远程推送 |
| 0 | 技术 Spike | `待开始` | 下一步优先执行，验证浏览器上下文、网络证据采集和最小宏闭环 |
| 1 | 工程骨架 | `待开始` | 建立 `FastAPI + React` 基础工程和本地联调 |

| 3 | 录制链路 MVP | `待开始` | 打通开始录制、结束录制、录制详情和会话管理 |
| 4 | 元数据审核 MVP | `待开始` | 打通 `MetadataDraft`、审核页与版本保存 |

## 1. 文档目的

本文档用于把首版开发过程拆成若干可控阶段，让项目始终按“先验证关键路径，再扩展功能”的方式推进。

它和 [首版实现计划](./首版实现计划.md) 的区别是：

- 本文档强调阶段推进和验收顺序；
- `首版实现计划` 更偏工程落地与模块任务清单。

## 2. 总体推进原则

- 始终优先打通用户闭环，而不是优先做技术层完备性。
- 每一阶段完成后都要能演示或验证。
- 前一个阶段未稳定，不进入下一个阶段。
- 对核心不确定点优先做技术 `Spike`。
- 首版不追求一步到位，先完成 `ActionMacro` 闭环。

## 3. 推荐阶段总览

```mermaid
flowchart LR
Spike[阶段0 技术Spike] --> Skeleton[阶段1 工程骨架]
Skeleton --> Models[阶段2 领域与存储]
Models --> Recording[阶段3 录制链路]
Recording --> Review[阶段4 审核链路]
Review --> Macro[阶段5 动作宏与执行]
Macro --> ImportExport[阶段6 导入导出]
ImportExport --> Hardening[阶段7 稳定性与收尾]
```

## 4. 阶段 0：技术 Spike

### 目标

先验证最不确定的技术链路，而不是马上搭完整工程。

### 要做的事

- 验证 `Playwright Python` 创建独立浏览器上下文；
- 验证页面导航和请求响应采集；
- 验证会话状态摘要是否能读取；
- 验证 `FastAPI` 与 `React` 本地联调；
- 验证一个最简单的宏执行闭环。

### 交付物

- 若干独立 PoC 脚本；
- 技术结论记录；
- 需要规避的坑清单。

### 验收标准

- 能打开指定 `URL`；
- 能抓到请求响应；
- 能拿到会话上下文摘要；
- 前后端本地联调成功；
- 至少一个最小动作链路被证明可行。

### 完成后再进入下一阶段

只有 `Spike` 证明主技术链可行，才值得开始正式工程搭建。

## 5. 阶段 1：工程骨架

### 目标

建立可运行的前后端工程基础。

### 要做的事

- 建立 Python 后端工程；
- 建立 React 前端工程；
- 建立基础目录结构；
- 建立本地运行脚本；
- 建立首页和基本导航；
- 建立 API 健康检查；
- 建立基础测试框架。

### 交付物

- 后端服务可启动；
- 前端开发服务器可启动；
- 首页能访问；
- 前后端联通。

### 验收标准

- 本地一条命令能分别启动前端和后端；
- 浏览器能访问首页；
- 前端可以调用健康检查接口。

## 6. 阶段 2：领域模型与存储骨架

### 目标

把对象模型和本地存储边界固定下来。

### 要做的事

- 实现核心领域模型；
- 建立 `SQLite` 元数据层；
- 建立证据文件区；
- 建立对象 ID 和状态机规则；
- 建立审核版本和动作版本结构；
- 建立初始化与迁移机制。

### 交付物

- 数据库结构；
- 本地目录结构；
- 基础仓储层；
- 版本关系说明。

### 验收标准

- 能创建和读取核心对象；
- 能持久化录制、动作和执行记录摘要；
- 能把大对象写入文件区并索引。

## 7. 阶段 3：录制链路 MVP

### 目标

完成“开始录制 -> 结束录制 -> 查看录制详情”的第一条业务闭环。

### 要做的事

- 建立浏览器会话管理；
- 建立录制任务创建与结束；
- 采集请求响应；
- 记录页面阶段；
- 记录会话状态摘要；
- 记录上传下载摘要；
- 建立录制状态与结束结果的 `SSE` 推送；
- 展示录制详情页；
- 建立录制列表页和会话管理页。

### 交付物

- 可操作的录制中心；
- 录制详情页；
- 会话管理页；
- 原始证据存储结果。

### 验收标准

- 用户可以输入 `URL` 开始录制；
- 用户结束录制后能看到请求索引和页面阶段；
- 能查看录制历史；
- 能看到会话与录制的关联关系。
- 录制中或录制结束状态可被前端实时感知。

## 8. 阶段 4：元数据审核 MVP

### 目标

完成“从录制结果到审核结果”的第二条闭环。

### 要做的事

- 建立 `MetadataDraft` 生成逻辑；
- 明确录制结束后只投递分析任务，不在录制链路中同步生成草案；
- 建立审核页；
- 建立元数据分析任务状态的 `SSE` 推送；
- 支持关键请求标注；
- 支持噪音请求标注；
- 支持参数来源确认；
- 支持字段说明编辑；
- 支持保存 `ReviewedMetadata` 新版本。

### 交付物

- 审核页面；
- 审核结果版本；
- 参数建议交互。

### 验收标准

- 用户可以从录制详情跳转到审核页；
- 用户可以完成一次审核并保存结果；
- 保存后可回看历史审核版本。
- 分析任务状态和完成结果可被前端实时更新。

## 9. 阶段 5：`ActionMacro` 与执行 MVP

### 目标

完成“从审核结果到动作执行”的第三条闭环。

### 要做的事

- 从审核结果生成 `ActionMacro`；
- 建立动作库列表和动作详情页；
- 建立参数定义；
- 建立执行入口；
- 建立执行日志与步骤状态的 `SSE` 推送；
- 在浏览器会话中执行宏；
- 展示执行日志和失败定位。

### 交付物

- 动作库；
- 动作详情；
- 执行中心；
- 执行记录页。

### 验收标准

- 用户可以选择一个宏并输入参数；
- 动作可以在浏览器中实际执行；
- 成功与失败都能被记录；
- 失败时能定位到具体步骤。
- 执行中的步骤变化和日志可以实时展示。

## 10. 阶段 6：导入导出与运行模式

### 目标

完成资料迁移能力，增强本地可用性。

### 要做的事

- 导出录制；
- 导出动作；
- 导出执行记录摘要；
- 导入资料包；
- 做清单预检查；
- 显示冲突与风险提示。
- 建立运行期前端构建产物挂载方案；
- 验证本地一体化单入口运行模式。

### 交付物

- 导入导出页；
- 资料包格式；
- 基础兼容检查机制。

### 验收标准

- 能导出一条录制或动作；
- 导入后能在系统中重新查看；
- 导入流程能清楚提示不迁移登录态。
- 后端可以挂载前端构建产物并作为本地单入口运行。

## 11. 阶段 7：稳定性与收尾

### 目标

把项目从“能跑”推进到“可持续开发”。

### 要做的事

- 补关键失败路径测试；
- 补基础日志与排错信息；
- 补启动说明；
- 补本地环境说明；
- 补开发模式与运行模式说明；
- 清理明显重复代码；
- 做一次完整端到端演示。

### 交付物

- 冒烟测试结果；
- 启动说明；
- 错误清单与剩余问题。

### 验收标准

- 新人可按说明启动；
- 至少一条完整业务闭环可稳定演示；
- 主要失败场景有清晰提示。

## 12. 每阶段的停顿点

建议在以下节点强制停下来复盘：

- `Spike` 完成后
- 录制链路打通后
- 审核链路打通后
- 执行链路打通后
- 导入导出完成后

每个停顿点都要回答：

- 当前最脆弱的链路是什么；
- 有没有必要提前引入更细的证据能力；
- 是否有阶段性范围失控；
- 下一阶段的前置条件是否真的满足。

## 13. 建议的实现顺序

严格建议按以下优先级执行：

1. 技术 `Spike`
2. 工程骨架
3. 领域模型与存储骨架
4. 录制链路
5. 审核链路
6. `ActionMacro` 执行
7. 导入导出
8. 稳定性收尾

不要一开始就先做：

- `BusinessAction` 完整闭环；
- 复杂动作编排；
- 细粒度 `DOM` 轨迹；
- 多人协作。

## 14. 与首版实现计划的关系

如果你要进入真正编码阶段，建议：

- 先按本文决定“先做哪个阶段”；
- 再参考 [首版实现计划](./首版实现计划.md) 的文件级落地建议；
- 编码时一次只执行一个阶段，不要把多个阶段混在一起。

## 15. 当前建议

如果马上进入开发，最合理的起点不是“直接搭全部系统”，而是：

> 先做 **阶段 0 技术 Spike**，确认浏览器上下文、网络证据采集和最小宏执行链路；  
> 然后再进入 **阶段 1 工程骨架** 和 **阶段 2 存储骨架**。

这样可以最大限度避免“工程搭起来了，但核心链路验证失败”的返工风险。
