# 首版实现计划

> **给后续执行者：** 建议按里程碑逐段实现，每完成一个里程碑就做一次独立审阅和可运行验证，不要一次性铺开所有模块。

**Goal:** 基于当前需求与技术方案，落地一个可在本机运行的 `WebToActions` 首版系统，完成“录制 -> 审核 -> 生成 `ActionMacro` -> 浏览器执行 -> 导出导入”的最小闭环。

**Architecture:** 首版采用 `FastAPI` 作为本地服务承载控制面，`React` 作为管理台，开发期前后端分离，运行期由本地服务挂载前端构建产物。浏览器控制层使用 `Playwright Python`，数据存储采用“`SQLite` 结构化索引 + 原始证据文件区”的双层模型。

**Tech Stack:** `Python 3.11+`、`FastAPI`、`Pydantic v2`、`SQLAlchemy 2`、`Alembic`、`Playwright Python`、`React`、`TypeScript`、`Vite`、`TanStack Query`、`React Router`、`Ant Design`、`SQLite`、`pytest`。

## 1. 实施策略

首版不建议按“技术层完备性”推进，而建议按“用户闭环 + 技术风险优先验证”推进：

1. 先证明浏览器链路可行；
2. 再搭稳定工程骨架；
3. 再固化领域和存储；
4. 再依次打通录制、审核、执行；
5. 最后补导入导出和稳定性。

这样做的原因是：这个项目最怕“工程壳子搭得很完整，但核心浏览器闭环并没有被真正验证”。

## 10. 测试策略

首版至少建立三层验证：

- 单元测试：领域对象、状态流转、参数映射；
- 集成测试：API、服务、存储、桥接层协作；
- 手工冒烟：真实浏览器场景下的录制、审核、执行闭环。

需要重点覆盖的风险点：

- 会话失效；
- 页面等待超时；
- 请求未发生；
- 审核版本保存冲突；
- 执行步骤失败；
- 导入包冲突。

## 11. 当前建议

如果马上进入编码，不建议直接做完整系统。  
建议顺序是：

1. 先做 [开发步骤拆解](./开发步骤拆解.md) 里的“阶段 0 技术 Spike”
2. 再做本文件的“里程碑 1 工程骨架”
3. 然后按里程碑顺序推进

这样最稳，也最能控制返工。
