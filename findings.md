# Findings

## 需求阶段结论

- 该产品不应被定义为“纯 API 自动化工具”，而应定义为“浏览器执行优先的动作系统”。
- 网络录制的核心价值在于帮助系统和人工理解用户行为、字段来源、依赖关系和参数语义。
- 人工审核不是补充流程，而是首版中的主流程之一。
- 若首版直接追求结构化业务动作自动生成，风险过高；应先让系统稳定沉淀可回放宏，再逐步抽象为业务动作。
- 对用户而言，登录态复用和执行稳定性比“完全脱离浏览器”更重要。
- 首版证据范围进一步收口为“网络请求/响应 + 页面导航 + 会话状态 + 文件上传下载”，细粒度 `DOM` 事件轨迹不再作为 MVP 必选项。
- 技术栈已正式从 `Java` 路线切换为 `Python 3.11+ + FastAPI + React + Playwright Python + SQLite`。
- 长任务通信方案已收口为 `SSE`，覆盖录制状态、元数据分析和动作执行日志。
- `MetadataDraft` 统一定义为录制结束后异步分析生成，而不是录制链路的直接同步产物。
- 主文档已按“产品文档”和“技术文档”分类迁移，相关引用统一指向新路径。
- `docs/技术文档/开发步骤拆解.md` 已被补强为可直接查看当前阶段状态的进度看板。
- 仓库开发前置已完成：本地 `git`、GitHub 远程、基础忽略规则、项目说明和环境样例文件均已就绪。
- 阶段 0 `Spike` 已证明 `Playwright` 独立浏览器上下文、请求/响应摘要采集和页面存储摘要采集是可行的。
- 最小前后端联调已验证通过：`FastAPI /api/health` 可通过 `Vite` 代理被前端页面访问。
- 浏览器 PoC 的一个关键边界已确认：必须在关闭浏览器上下文前完成 `storage_state()` 和页面存储快照采集，否则会触发 `Event loop is closed`。
- 开发环境存在真实端口占用风险，后续正式工程需要显式处理后端端口与前端 dev server 端口回退问题。
- 浏览器运行策略已收口为“双轨制”：默认使用 `Playwright Chromium` 作为稳定基线，同时支持本机已安装 `Chrome` 作为可选通道。
- “使用本机 Chrome” 不等于“自动复用当前日常 Chrome 的 Profile / 登录态”，首版仍以隔离上下文为主。
- `docs/技术文档/开发步骤拆解.md` 已补充新 `session` 接力说明，可作为后续新会话直接启动待开始阶段开发的主执行文档。
- `阶段 1` 已完成正式工程化收口：后端已具备统一配置入口，前端已具备正式路由壳子与一级导航占位。
- `Python 3.11+` 已从文档要求收口为工程基线，后端运行和验证不再以本机 `Python 3.10.0` 为兼容目标。
- 本地开发启动入口已统一到 `scripts/dev/start_backend.sh` 与 `scripts/dev/start_frontend.sh`，新 `session` 可直接复用。
- 前端在 `阶段 1` 即补入了最小 `Vitest + React Testing Library` 基线，可支持后续页面骨架演进时的回归验证。
- 当前 `start_frontend.sh` 仍沿用 `Vite` 默认端口回退策略；若 `5173` 被占用，开发服务器会自动切到其他端口，使用时应以终端输出为准。
- 默认 `WEBTOACTIONS_DATA_DIR` 已锚定到仓库根目录；对外健康契约仍保持相对展示值 `.webtoactions`，从而兼顾运行稳定性与既有文档口径。
- `阶段 2` 应以后端为主，不需要独立前端 agent；现有前端壳子对领域模型、SQLite 和文件对象区没有直接实现要求。
- `BusinessAction` 在阶段 2 只保留模型与存储留位，不推进完整仓储、服务或 API。
- `BrowserSession`、`Recording`、`ExecutionRun` 等关键对象需要在阶段 2 显式收口状态枚举与合法转移，而不是仅保留宽松占位。
- `阶段 2` 的最自然拆分是：先冻结领域契约，再并行实现 `SQLite/Alembic` 与文件对象区 bootstrap，最后串行整合仓储与启动初始化。
- 阶段 2 已把版本化对象统一为“稳定工件 ID + version + previous_version”口径，后续 `SQLite` 表结构、执行记录引用和导出包清单都可沿这套语义继续扩展。
- 阶段 2 的领域对象已统一采用不可变模型与返回新实例的转移方式，避免运行期直接改字段绕过状态机和版本链约束。
- `RequestResponseRecord` 已从“直接承载原始请求体/响应体”收口为“轻量索引摘要 + blob key 引用”；后续大对象应落到文件对象区，而不是塞进 `SQLite`。
- `SQLite` 元数据层与文件对象区边界已经固定：结构化索引进入 `app.db`，请求体、响应体、会话大对象、文件传输对象和导出包本体进入 `.webtoactions/` 文件区。
- `FastAPI` 生命周期现已承担阶段 2 的基础设施初始化职责：应用启动时自动创建 `.webtoactions/`、执行 Alembic 升级，并把最小仓储挂到 `app.state`。
- 最小仓储闭环已验证通过：当前至少可以在不依赖前端的情况下创建并读取一个 `BrowserSession + Recording + MetadataDraft + ReviewedMetadata` 聚合。
- 阶段 3 录制链路继续沿用“隔离 `Profile` + 本地服务主控”的策略，浏览器桥接层采用 `Playwright` 持久化上下文，录制时把会话状态保留在会话目录中，而证据索引落在 `SQLite + 文件对象区`。
- 阶段 3 的录制运行态采用“内存中的活动录制 + 结束时持久化聚合”模式：录制中通过 `SSE` 推送请求数、页面阶段数和当前页面，结束录制后再把请求体/响应体 blob、页面阶段、会话快照和文件传输摘要一次性落盘。
- 录制状态流的前后端契约已收口为 `/api/recordings/{id}/events`，并支持先返回最新快照、再继续推送后续变化；前端录制详情页只需要消费这一条 `SSE` 即可实时刷新关键计数。
- 阶段 3 前端暂不引入额外状态库，直接基于 `fetch + EventSource + React Router` 完成录制列表、新建录制、录制详情和会话管理四个页面，足够支撑后续阶段 4 的审核入口接入。
- 阶段 3 完成后，健康检查契约中的 `phase` 已升级为 `stage3`，仓库下一步应进入 `阶段 4：元数据审核 MVP`，而不是继续补录制页的横向扩展能力。
- 阶段 4 的 `MetadataDraft` 首版没有接入真实 `LLM`，而是先采用“确定性规则分析 + 结构化建议”的策略：直接复用录制证据中的请求体 JSON、页面阶段和请求关联关系生成候选参数与动作片段，优先保证可验证和可回放。
- 阶段 4 的分析任务运行态采用“内存任务运行器 + `SSE` 快照”模式：录制结束后自动投递分析任务，审核页也可按需触发补跑；任务状态只保留在运行内存中，分析结果仍持久化到 `SQLite + 文件对象区`。
- 为了真正支持“关键请求 / 噪音请求”双向标注，`ReviewedMetadata` 在阶段 4 新增了 `noise_request_ids` 持久化字段，而不是把噪音结果隐式推导成“非关键请求”。
- 阶段 4 的前端审核入口收口为 `/review/:recordingId` 单录制审核页，而不是先做审核列表页；在托管推进场景下，这样可以最短路径打通“录制详情 -> 审核 -> 保存版本”的主闭环。
- 阶段 4 的审核接口必须限制在录制完成之后才能启动分析；同时保存 `ReviewedMetadata` 时需要校验请求 ID、页面阶段 ID 与录制聚合的一致性，并拒绝 `key/noise` 重叠的非法输入。
- 阶段 4 完成后，健康检查契约中的 `phase` 已升级为 `stage4`，仓库下一步应进入 `阶段 5：ActionMacro` 与执行 MVP，而不是继续横向扩写审核页。
